6. PROBE SELECTION
================================================================================

By operation-type inversion. v0.1 uses generic direct_phenomenology
for SIMILE; refinement deferred to v0.2 of this work.

  ┌──────────────────────┬──────────────────────────────────┐
  │     Locked opcode    │      Probe operation type        │
  ├──────────────────────┼──────────────────────────────────┤
  │ OBSERVE_*            │ direct_phenomenology             │
  │ INTROSPECT_*         │ grounded_observation             │
  │ SIMILE / TRANSLATE   │ direct_phenomenology  (v0.1)     │
  │ DIALECTICAL          │ substitution                     │
  │ ACTION               │ direct_phenomenology             │
  │ OTHER/UNKNOWN        │ direct_phenomenology  (default)  │
  └──────────────────────┴──────────────────────────────────┘

Within selected category: pick LRU probe. If LRU was used <4h ago,
escalate to next category in mapping.

If all categories exhausted in last 4h: trigger operator-prompt
mode (see Section 9 Q7) — do not auto-generate probes.

7. OPERATOR APPROVAL (Phase B)
================================================================================

7.1 Approval flow:

  1. Write row to auto_probe_log with status='pending_approval'
  2. Send Telegram message to @Nex_4bot:
       "GROOVE DETECTED [auto_probe v0.1]

        Severity: X.XX
        Template: '<truncated text>'
        Sustained: N fires
        Locked opcode: <NAME>

        Suggested probe: [<category>]
        '<probe_text>'

        Reply /approve, /reject, or do nothing within 60s."

  3. Wait up to 60 seconds polling for reply
  4. On /approve: status → 'approved', proceed to fire
  5. On /reject: status → 'rejected_by_operator', set 4h cooldown,
     log reply text in operator_reply field
  6. On 60s timeout: status → 'would_have_approved', NO FIRE.
     Log captures what would have happened. Operator reviews
     retroactively.

CHANGE FROM v0.1: timeout no longer auto-approves and fires.
This preserves operator-in-loop discipline through Phase B.
Auto-firing on timeout is a Phase C behavior.

7.2 In Phase A: no Telegram, no firing. Log only with status='observation'.
7.3 In Phase C: no approval step. Fire immediately on trigger.

================================================================================
8. FIRING PATH
================================================================================

Uses existing ProbeRunner.run_probe() without modification.

Pseudocode:

  result = probe_runner.run_probe(
      category=selected_category,
      probe_text=selected_probe_text,
      notes=f"auto_probe:groove_breaker severity={severity:.2f}",
  )

After run_probe() returns:

  1. If result["response_text"] empty or starts with "[ERROR":
     mark as 'error', do NOT inject. Log error details.
  2. On success: inject into fountain.recent_thoughts with
     source='auto_probe' tag, mechanism per Section 4 prerequisite
     (weight or alternative method to be determined).
  3. Update auto_probe_log: status='fired', probe_id,
     response_preview (first 200 chars).

The injection seeds; it does not force. Fountain's normal selection
logic decides what to use on next tick.

11. ROLLOUT PLAN
================================================================================

Phase A — Observation (3 days)
  PREREQUISITE: Recent_Thoughts_Selection_Notes.txt committed

  - groove_breaker.py wired in, checks trigger conditions every tick
  - Logs status='observation' when trigger fires
  - No Telegram, no firing
  - Goal: validate trigger conditions fire at sensible rate (0-3/day)
  - Exit: ≥2 logged events, rate not excessive

Phase B — Operator-Approved (10 days)
  - Approval flow per Section 7
  - Operator silence → log only, no fire (changed from v0.1)
  - Daily review of auto_probe_log
  - Adjust thresholds based on data
  - GUI manual trigger button live (from Q3)
  - Goal: confirm probe selection coherent; groove-break rate >50%
  - Exit: ≥5 grooves attempted via approval, >50% broken

Phase C — Autonomous (after Phase B success)
  - No approval step
  - Hard cooldown and backoff still enforced
  - GUI kill-switch in cockpit panel
  - Auto-fires on trigger conditions

================================================================================

  - Groove detector false positives (mitigated by Phase A tuning)

================================================================================
13. WHAT THIS DOES NOT DO
================================================================================

  - Does not modify substrate (LLM, prompt builder, retrieval)
  - Does not change crystallizer
  - Does not alter mode-state
  - Does not add operation types
  - Does not change 6-mode taxonomy

Amplification of existing capability (probes), not architectural
modification. Within NEX's own logic per Phase 2 discipline.

================================================================================
14. SUCCESS CRITERIA
================================================================================

After 2 weeks of Phase B operation:

  PRIMARY:
   - Mean lock duration < 1 hour (vs ~14h baseline)
   - 90%+ of severe grooves cleared by first or second probe
   - Zero false positives disrupting healthy register runs

  SECONDARY:
   - New fountain_insight beliefs/day increase
     (baseline: ~1/day during locks, ~10/day healthy)
   - Operator approval rate >70%
   - Operator silence rate (would_have_approved) <30%

  FAILURE INDICATORS:
   - Probe-loop (probes themselves grooved): >2 same probe in 24h
   - High operator silence: >50% events as would_have_approved
     (suggests timeout window too tight or operator overwhelmed)
   - Trigger thrashing: >5 events/day (threshold too sensitive)
