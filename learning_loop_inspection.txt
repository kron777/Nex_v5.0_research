1. WHERE COLLISION_GRADER PRODUCES GRADES
-----------------------------------------
File: theory_x/diversity/grader.py
Function: CrossbreedGrader.grade(), lines 48–80

How it works:
  - Takes child_id, parent_a_id, parent_b_id (the synergized belief + its parents)
  - Embeds all three beliefs (via theory_x/diversity/embeddings.py)
  - Computes three components:
      input_dist  = distance(embed_a, embed_b)        — how different the parents are
      output_dist = min(dist(child,a), dist(child,b)) — how novel the child is
      rarity      = 1/(1 + n_prior_same_branch_crosses / 5.0) — cross-branch novelty
  - Weighted sum: grade = 0.40*input_dist + 0.35*output_dist + 0.25*rarity
  - Writes row to collision_grades table (belief_id, parent_a_id, parent_b_id,
    input_distance, output_distance, rarity, grade, grader_version, graded_at)
  - Returns grade as float

Called from TWO places:
  A) synergizer.py:104 — inline after synergized belief is written
     (active code path, responsible for all 237 rows in collision_grades)
  B) loop.py:71 — DiversityLoop.grade_synergy() method
     NOTE: grade_synergy() is NEVER CALLED from anywhere outside the class.
     It is a dead method. The synergizer uses its own inline path (A), not
     the loop path. This is not the bug but worth noting.

What the synergizer does with the grade right now (synergizer.py:98-108):

5. RETRIEVAL QUERY — HOW BOOST FLOWS INTO RETRIEVAL
-----------------------------------------------------
File: theory_x/stage6_fountain/generator.py
Function: FountainGenerator._retrieve_context_beliefs(), lines 544–627

Own-content retrieval query (lines 561–568):

  SELECT b.id, b.content, b.source, b.tier, b.confidence, b.created_at,
         COALESCE(bb.boost_value, 1.0) AS boost_value
  FROM beliefs b LEFT JOIN belief_boost bb ON b.id = bb.belief_id
  WHERE b.source IN (<own_content_sources>)
  ORDER BY (b.created_at * COALESCE(bb.boost_value, 1.0)) DESC LIMIT ?

Boost mechanics:
  - belief_boost is LEFT JOINed on belief_id
  - COALESCE(bb.boost_value, 1.0) defaults to 1.0 for un-boosted beliefs
  - ORDER BY multiplies created_at × boost_value
  - A boost_value of 1.35 (grade=0.35) makes a belief rank as if it is 35%
    newer than it actually is — pushing it up in the retrieval window
  - boost_value is also recorded in retrieval_manifest and logged to
    fountain_retrieval_log(boost_value) for each fire

Seed retrieval (lines 571–581): does NOT use belief_boost at all.
  Uses ORDER BY RANDOM() — seeds are not eligible for boost.

Boost flows correctly once applied. The retrieval query is correct.
The issue is solely that no boosts have ever been applied.
