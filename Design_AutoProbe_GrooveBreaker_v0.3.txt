5. TRIGGER CONDITIONS
================================================================================

All four conditions must hold simultaneously:

  1. Active groove_alert with severity >= 0.80
     Query: SELECT * FROM groove_alerts
            WHERE severity >= 0.80
            AND detected_at > strftime('%s','now') - 1800
            ORDER BY detected_at DESC LIMIT 1
     (1800s = 30 min freshness window)

  2. Groove sustained: at least 8 fountain_events fired since
     the alert's first detection
     Query: SELECT COUNT(*) FROM fountain_events
            WHERE ts > <alert.first_detected_at>
            AND hot_branch NOT IN ('voice_fallback','quiescent')

  3. Cooldown elapsed: last auto_probe_log entry > 30 min ago
     Query: SELECT MAX(ts) FROM auto_probe_log
     (Compare to current time, require > 1800s diff)

  4. No pending approval: no row in auto_probe_log with
     status='pending_approval'

CHANGE FROM v0.2: Trigger 2 was previously "consecutive_same_
template >= 8" assuming a template_hash was tracked per fire.
Investigation found no such hash exists. The groove detector
itself uses bigram overlap across a 20-belief window. Reformulated
to use detector's own alert as the gate, plus a sustain count of
fires since the alert started.

6. PROBE SELECTION
================================================================================

Same as v0.2. By operation-type inversion. v0.1 uses generic
direct_phenomenology for SIMILE; refinement deferred.

  ┌──────────────────────┬──────────────────────────────────┐
  │     Locked opcode    │      Probe operation type        │
  ├──────────────────────┼──────────────────────────────────┤
  │ OBSERVE_*            │ direct_phenomenology             │
  │ INTROSPECT_*         │ grounded_observation             │
  │ SIMILE / TRANSLATE   │ direct_phenomenology  (v0.1)     │
  │ DIALECTICAL          │ substitution                     │
  │ ACTION               │ direct_phenomenology             │
  │ OTHER/UNKNOWN        │ direct_phenomenology  (default)  │
  └──────────────────────┴──────────────────────────────────┘

LRU within category. If LRU < 4h ago, escalate to next
category. If all exhausted in 4h, operator-prompt mode.

To extract dominant_opcode from the groove alert, examine the
alert's matched_text or the most recent N fountain_events
preceding the alert and classify via existing register patterns.

8. FIRING PATH
================================================================================

Uses existing ProbeRunner.run_probe() without modification.

Pseudocode:

  result = probe_runner.run_probe(
      category=selected_category,
      probe_text=selected_probe_text,
      notes=f"auto_probe:groove_breaker severity={severity:.2f}",
  )

After run_probe() returns:

  1. If result["response_text"] empty or starts with "[ERROR":
     mark as 'error', do NOT write belief. Log error details.

  2. On success: write the probe response as a belief in
     beliefs.db:

       INSERT INTO beliefs (
         content,
         tier,
         source,
         created_at,
         updated_at,
         provenance,
         boost_value
       ) VALUES (
         <result["response_text"]>,
         6,
         'auto_probe',
         strftime('%s','now'),
         strftime('%s','now'),
         <JSON: groove_signal_id, severity, probe_id, dominant_opcode>,
         1.0
       )

     With source='auto_probe' (in _OWN_CONTENT_SOURCES per 4b)
     and created_at=now, this belief wins the top slot in the
     next fountain context retrieval (ORDER BY created_at *
     boost_value DESC). Boost stays at default 1.0 — recency
     alone is sufficient.

  3. Update auto_probe_log: status='fired', probe_id,
     belief_id (the newly inserted belief),
     response_preview (first 200 chars).

CHANGE FROM v0.2: v0.2 said "inject into fountain.recent_thoughts
with weight=1.5". Investigation found no recent_thoughts pool
exists. Replaced with belief insert. The boost=1.5 guess is
unnecessary — created_at=now ranks above all older content with
boost=1.0.

The fountain's normal selection logic decides what to use on
next tick. We seed; we do not force.

10. NEW DB TABLE: auto_probe_log
================================================================================

  CREATE TABLE IF NOT EXISTS auto_probe_log (
      id                INTEGER PRIMARY KEY AUTOINCREMENT,
      groove_alert_id   INTEGER,          -- FK into groove_alerts
      groove_severity   REAL,
      dominant_opcode   TEXT,
      sustained_fires   INTEGER,          -- fires since alert detected
      selected_category TEXT,
      probe_id          INTEGER,          -- FK into probes table
      belief_id         INTEGER,          -- FK into beliefs (the injected row)
      status            TEXT,
        -- 'observation'
        -- 'pending_approval'
        -- 'approved'
        -- 'rejected_by_operator'
        -- 'would_have_approved'
        -- 'fired'
        -- 'failed_to_break'
        -- 'groove_broken'
        -- 'error'
      operator_reply    TEXT,
      response_preview  TEXT,
      ts                REAL
  );

Lives in beliefs.db alongside probes table.
Written by auto_probe/groove_breaker.py via beliefs Writer.

CHANGE FROM v0.2: groove_signal_id renamed to groove_alert_id
(matches actual table name). Added belief_id field for
traceability of the injected belief. Replaced consecutive_count
field with sustained_fires (fires since detection).
