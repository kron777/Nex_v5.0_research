G. RETRIEVAL PATH — WHY voice_ok=False PERSISTS
======================================================================

VOICE CALL ARCHITECTURE:

  The fountain generator calls self._voice.speak(VoiceRequest(...)) once per fire.
  The exception is caught at the fire level:

    try:
        resp = self._voice.speak(VoiceRequest(...), beliefs=None)
        thought = resp.text.strip()
    except Exception as e:
        voice_ok = False
        logger.warning("Fountain: voice unreachable, using sense fallback: %s", e)
        [fallback to raw sense event]

  There is no:
    - Retry with backoff within a single fire
    - Session-level reconnection logic between fires
    - Health check before calling self._voice.speak()
    - Circuit breaker that backs off after N consecutive failures

  Each fire independently attempts the LLM call. If the underlying voice
  transport (HTTP endpoint, socket, or subprocess) is in a broken state, every
  fire throws, every fire catches, every fire stores raw JSON.

H3. WHY NO SELF-RECOVERY IN VOICE_FALLBACK MODE

  The voice_fallback path has no recovery logic by design. The catch block at
  the fire level is a graceful degradation: store something, keep the loop alive,
  log the error. The next fire in 120 seconds will independently attempt the LLM
  call again. If the voice transport recovers (endpoint becomes reachable, subprocess
  restarts), the next LLM call succeeds and the system silently resumes normal output.

  This means voice_fallback is self-healing — but only if the underlying transport
  issue resolves. The fallback period length equals the transport outage duration.
  From the evidence (voice_fallback persists through at least fev=1215, ~14:00 UTC),
  the transport outage lasted approximately 2+ hours after restart.

H4. READINESS CALCULATION AT BREAK POINT

  Pre-break fires all show readiness=0.700. With fires every ~120s:
    2 hot branches (systems + crypto)  → 2 × 0.3 = 0.6
    beliefs > 20 (beliefs ≫ 20)        → +0.1
    elapsed ≥ 600s                     → +0.0  (fires every 120s, elapsed < 600)
    total readiness                    =  0.7  (exactly at threshold)

  This is the floor state: readiness stays at 0.7 and never rises above it
  because the 600s elapsed bonus never triggers when fires happen every 2 min.

  First post-restart fire (fev=1197) shows readiness=0.9:
    The 32.4-minute gap meant elapsed ≥ 600s → +0.2
    0.6 (branches) + 0.1 (beliefs) + 0.2 (elapsed) = 0.9
    Subsequent fires immediately drop back to 0.7 (elapsed resets).
